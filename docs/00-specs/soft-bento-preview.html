Principles

Four rules that govern every visual decision in Soft Bento. When in doubt, defer to these.

  • Calm modernism, not minimalism. Generous rounded surfaces, soft cream backgrounds, breathing room. The mood is contemporary Tokyo/Vancouver crossover branding — warm, considered, not stark.
  • Semantic color discipline. Jade means have. Persimmon means buy (and primary accent). Apricot means alternative / secondary action surface. Shoyu (near-black) means commit / primary button. Don't use these colors decoratively — every appearance carries meaning.
  • One type system. DM Sans for Latin, Noto Sans SC for CJK. No serifs anywhere. JetBrains Mono only for terse cooking-instruction reference blocks (Saved Recipe, Saved Plan detail).
  • Bilingual is a first-class state, not a toggle. Chinese names for dishes appear inline with English at all times. Functional UI (buttons, labels, hints) stays in English only. See §5.

Bilingual rules

When CJK appears and when it doesn't. Translation is for cultural information, not redundancy.

CJK appears on:

  • Dish names in dish cards (.dish-cjk)
  • Alternative names in the alts panel (.alt-cjk)
  • Recipe titles in saved content (when those screens are designed)
  • Ingredient names only when culturally specific (e.g., 五花肉 for pork belly when teaching the term)

CJK does NOT appear on:

  • Functional buttons (Save plan, Build list, try another, →)
  • Section labels (try instead, A note to the agent)
  • Body copy / deck text
  • Stat chips, meta strings, hints
  • Status bar, footer

Why: bilingual users navigating English UI fluently don't need every word translated. They need help with names of things they may not recognize. Translating buttons clutters the interface and patronizes the user. Translating dish names is informative — it's how the two user groups (Vancouver immigrants exploring Western food, locals exploring Asian food) actually search and recognize.

Font fallback: the --cjk stack is 'Noto Sans SC', 'DM Sans', sans-serif. Latin characters inside CJK strings render in Noto Sans SC (which has full Latin coverage), so mixed strings look consistent. Don't manually swap fonts mid-string.

Handoff notes

Instructions for Claude Code or any developer building from this reference.

Implementation order. Build in stages, validate at each:

  • Stage 1 — static HTML/CSS. Generate the 6 remaining screens (Home, Clarify, Grocery, Saved Plan, Saved Recipe, Saved List) as static HTML using only the tokens, components, and rules in this document. No JS, no framework. Quick visual review pass.
  • Stage 2 — React/TS components. Convert validated HTML into typed React components with mock props. Still no backend.
  • Stage 3 — mock data + state machine. Wire the IDLE → LOADING → STREAMING → COMPLETE states from architecture-spec-v2.md §11 with hardcoded responses. Validate full click-through.
  • Stage 4 — real AI layer. Connect to /chat SSE endpoint with typed event handling per ai-layer-architecture-v2.md §3.

What to copy from this doc:

  • The full :root CSS variable block (§2 colors + §3 type stacks). Copy as-is.
  • Component class names exactly as shown — they're the contract between screens.
  • The Recipes render as the visual quality bar.

What NOT to do:

  • Don't introduce new font families. DM Sans + Noto Sans SC + JetBrains Mono is the complete set.
  • Don't introduce new colors outside the token list. If a screen needs a new accent, add it to this doc first.
  • Don't translate functional UI to Chinese. See §5.
  • Don't use serifs anywhere — Soft Bento is fundamentally modern.
  • Don't swap semantic colors (jade ≠ persimmon). They carry meaning.

New components needed for other screens (to design when reaching each):

  • PCV category row (Clarify) — status icon (ok/warn/bad) + label + items text
  • Quick-start chip (Home) — pill with template label
  • Fridge recall banner (Home) — soft callout with dismiss
  • Checklist row (Grocery, Saved List) — checkbox + item + sub + remove
  • Store section header (Grocery, Saved List) — small caps label
  • Expandable recipe row (Saved Plan) — tap-to-reveal cooking steps
  • Recipe detail block (Saved Plan, Saved Recipe) — JetBrains Mono terse instructions
  • Sidebar menu (Home overlay) — three-section saved content list

Reference files:

  • wireframe-v2.html — structure and interaction source of truth (follow faithfully, restyle visually)
  • product-spec-v2.md — what each screen does and why
  • architecture-spec-v2.md §11 — frontend architecture and state machine
  • ai-layer-architecture-v2.md §3 — SSE event types for stage 4
